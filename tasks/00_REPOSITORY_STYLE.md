# PMM repository style guide

Этот документ задаёт **единый стиль** для репозитория, документации, комментариев и внутренних инженерных решений.

Он нужен не как литературный стандарт, а как механизм удержания репозитория в компактном и рабочем состоянии.

## 1. Общий принцип

В репозитории должен существовать **один канон на один вопрос**.

Нельзя одновременно поддерживать:
- несколько равноправных “главных” описаний одной архитектуры;
- несколько параллельных объяснений одной и той же семантики;
- несколько слоёв совместимости без подтверждённой необходимости;
- комментарии, документацию и код, противоречащие друг другу.

## 2. Стиль репозитория

### 2.1. Что считается first-class частью репозитория

First-class считаются только:
- `include/`
- `single_include/`
- `tests/`
- `benchmarks/`
- `docs/`
- `demo/`
- `examples/`
- `scripts/`
- root-level build/release files
- `README.md`

Все остальные файлы и каталоги должны быть либо:
- явно нужны прямо сейчас;
- перенесены в более подходящее место;
- удалены.

### 2.2. Правило “одна активная поверхность”

Для каждой инженерной темы должен быть ровно один активный вход:
- один индекс документации;
- один основной архитектурный документ;
- один основной документ по семантике;
- один основной документ по recovery/validation;
- один основной документ по storage seams.

Исторические материалы не должны участвовать в основной навигации.

### 2.3. История живёт в Git, а не в канонических docs

В канонических docs не нужны:
- длинные перечни старых issue;
- журналы “что было сделано в версии X”;
- пошаговая история развития мысли;
- ссылки на закрытые промежуточные решения.

Это всё уже есть в:
- Git history;
- issues / pull requests;
- release notes / changelog.

## 3. Стиль документации

### 3.1. Документация должна описывать только:
- текущее состояние;
- целевое состояние;
- инварианты;
- границы ответственности;
- допустимые и недопустимые сценарии.

### 3.2. Каждый canonical doc должен иметь одинаковый каркас

Обязательные разделы:
1. `Purpose`
2. `Scope`
3. `Core model / invariants`
4. `Rules / contracts`
5. `Out of scope`
6. `Related docs`

### 3.3. Документация не должна:
- дублировать README;
- пересказывать код построчно;
- хранить исторические фазы проекта;
- содержать issue-ссылки в теле нормативного текста.

### 3.4. Язык документации

Допустим русский как основной язык репозитория. Важнее единообразие, чем двуязычность.

Если документ остаётся русским, то:
- заголовки файлов именуются по-английски;
- термины в тексте используются стабильно;
- один и тот же термин не меняет перевод между документами.

## 4. Стиль комментариев в коде

### 4.1. Комментарий нужен только если он делает одно из трёх
- фиксирует инвариант;
- объясняет небанальную причину решения;
- предупреждает о риске поломки / ловушке.

### 4.2. Комментарий не нужен, если он:
- просто пересказывает код;
- повторяет имя функции или поля;
- объясняет очевидную механику языка C++;
- ссылается на старый issue вместо описания сути.

### 4.3. Разрешённые типы комментариев
- **Invariant** — что обязано быть истинным всегда
- **Persistence contract** — что должно переживать reload / relocation
- **Safety note** — что нельзя нарушать без UB/corruption
- **Design note** — короткое объяснение, почему решение именно такое

### 4.4. Запрещённые типы комментариев
- “сделано в issue #...”
- “раньше было так-то”
- “временно оставлено”
- “возможно удалить позже”
- многострочные narrative-комментарии без инварианта

## 5. Стиль кода

### 5.1. Одна возможность — один путь исполнения
Если capability уже реализована, в репозитории не должно быть двух полноценных
параллельных путей без строгой причины.

### 5.2. Public API тонкий, внутренности изолированы
- публичная поверхность минимальна;
- общий код и helper-логика уходят в `detail`;
- внутренние слои не притворяются публичными abstractions без необходимости.

### 5.3. Совместимость не является добром по умолчанию
Compatibility layer оставляется только если выполнено хотя бы одно условие:
- это публичное обещание API;
- это реально используемый platform-specific path;
- это подтверждено активными тестами;
- удаление сейчас ломает важный внешний use case.

Иначе совместимость удаляется.

### 5.4. Header-only не оправдывает разрастание
Header-only — это ограничение поставки, а не оправдание:
- длинных цепочек include;
- дублирования шаблонной логики;
- трудно читаемых слоёв-обёрток;
- “универсальности” без реальной пользы.

## 6. Стиль задач

Каждая новая задача в `tasks/` должна быть:
- узкой;
- проверяемой;
- с чёткими deliverables;
- с явными acceptance criteria;
- с явным out of scope.

Нельзя формулировать задачу как “улучшить что-нибудь вокруг ...”.

## 7. Правило принятия изменений

Изменение считается хорошим, если одновременно:
- уменьшает когнитивную нагрузку;
- не размывает границы PMM;
- не плодит новые документы без удаления старых;
- не добавляет новую compatibility-ветку без тестовой причины;
- улучшает проверяемость инвариантов.

Если изменение делает код или docs длиннее, оно должно явно окупаться:
- новым инвариантом,
- новым safety contract,
- новым seam,
- или заметным упрощением дальнейшего развития.
