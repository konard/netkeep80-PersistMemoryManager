# Delete compatibility code and dead paths

## Purpose

Найти и удалить код, который остаётся в репозитории только:
- ради старой совместимости;
- ради переходного периода, который уже закончился;
- ради гипотетических use cases без тестов и пользователей;
- или просто потому, что его никто не решился удалить.

## Why now

Compatibility-код почти всегда дорог:
- увеличивает ветвление;
- скрывает настоящий активный путь исполнения;
- усложняет validation и recovery;
- делает API и внутреннюю структуру визуально тяжелее.

Если его не вычистить сейчас, задачи 08–10 будут строиться уже на лишней основе.

## Deliverables

1. `docs/compatibility_audit.md` — таблица всех compatibility/legacy путей с решениями.
2. Удаление всех legacy paths, не прошедших критерий сохранения.
3. Обновлённые tests/docs после удаления.
4. Явный список допустимых compatibility seams, которые остаются.

## Preservation criteria

Compatibility layer можно оставить только если выполнено хотя бы одно условие:
- это обещанный public API, который реально стоит поддерживать;
- это platform-specific path, подтверждённый тестами;
- это migration seam, без которого ломается текущий supported workflow;
- это low-cost shim с высокой практической ценностью.

Если ни одно условие не выполняется, путь удаляется.

## Work breakdown

### A. Сделать audit всех подозрительных слоёв
Проверить:
- старые alias / wrappers;
- fallback paths;
- переходные adapter-слои;
- deprecated API;
- старые runtime branches;
- “на всякий случай” сохранённые helper-ветки.

### B. Отделить public compatibility от internal leftovers
Нужно отдельно отметить:
- что реально требуется внешнему пользователю;
- что осталось только внутри кода.

Internal legacy не должен сохраняться “ради осторожности”.

### C. Удалить неподтверждённые ветки
Удалять всё, что:
- не покрыто тестами;
- не упоминается как active API;
- не входит в canonical docs;
- не даёт актуальной платформенной ценности.

### D. После удаления упростить contracts
После каждого удаления нужно:
- сузить соответствующий кодовый контракт;
- убрать лишние ветвления;
- обновить docs и tests.

## Acceptance criteria

- Все compatibility/legacy paths перечислены в одном audit-документе.
- Для каждой такой ветки принято решение keep/delete.
- Неостановленные legacy paths имеют явную аргументацию.
- Неподтверждённые compatibility layers удалены.
- После задачи публичный и внутренний путь исполнения стал заметно прямее.

## Out of scope

- новая функциональность;
- rewrite allocator-а;
- тестовая матрица задач 09;
- storage seams задач 10.
