# PMM compactification backlog — index

Этот пакет заменяет старый backlog 01–10 новым планом для **промежуточной компактификации** `PersistMemoryManager`.

Смысл этого этапа:
- сначала **сжать и упорядочить** репозиторий;
- затем **зафиксировать канонические инварианты** уже реализованной архитектуры;
- только после этого закрыть оставшиеся большие блоки: **validation, test matrix, storage seams**.

Новый порядок намеренно другой. Старый план был архитектурно верным, но он почти не выделял саму борьбу с раздуванием как отдельную работу. Здесь она поставлена в начало.

## Принцип порядка

1. Сначала фиксируем, **каким должен быть компактный репозиторий** и что в нём считается каноном.
2. Потом убираем **исторический шум**: фазовые документы, ссылочные комментарии, legacy/compatibility-код, лишние слои.
3. Затем замораживаем **ядро инвариантов**, уже появившееся после старых задач 01–07.
4. И только после этого доводим до жёсткого состояния:
   - verify/repair contract,
   - pointer and block validation,
   - reload/corruption test matrix,
   - storage seams и mutation ordering.

## Как новый план соотносится со старым

### Старые 01–06
Не переоткрываются как отдельные большие архитектурные эпики. Их результат должен быть:
- проверен на непротиворечивость;
- собран в канонический набор документов;
- очищен от исторических наслоений;
- отражён в компактном коде и инвариантах.

Эту работу закрывают новые задачи **01–06**.

### Старая 07
Сохраняется по смыслу, но переводится из формата “ввести verify/repair” в формат
**“зафиксировать окончательный контракт verify/repair и диагностик после компактификации”**.

Эту работу закрывает новая задача **07**.

### Старые 08–10
Сохраняются как отдельные финальные блоки и входят в новый план практически напрямую:
- новая **08** = pointer/block validation;
- новая **09** = reload/corruption/property test matrix;
- новая **10** = storage seams + mutation ordering.

## Что этот этап должен дать на выходе

После выполнения пакета репозиторий должен стать:
- компактнее по структуре;
- тише по комментариям;
- короче по внутреннему коду;
- жёстче по инвариантам;
- понятнее по документам;
- безопаснее для дальнейшего развития `pjson_db` и верхних слоёв.

## Что не является целью этого пакета

- новая продуктовая функциональность;
- расширение PMM в сторону `pjson`-семантики;
- написание “истории проекта”;
- сохранение любого legacy-кода только ради “а вдруг пригодится”.
