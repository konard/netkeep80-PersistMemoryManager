# Define the target repository shape and deletion policy

## Purpose

До любых точечных рефакторингов нужно определить **целевую форму репозитория**:
что в нём является каноническим, что вторичным, что подлежит архивированию, а что удалению.

Без этого любая “компактификация” превратится в локальную уборку без общего критерия.

## Why now

Сейчас репозиторий уже содержит:
- полноценную библиотеку;
- тесты и бенчмарки;
- набор архитектурных документов;
- фазовые/исторические документы;
- несколько разных слоёв описания одного и того же.

Новый код и новые docs уже начинают садиться поверх старых, а не вместо них.
Нужно сначала остановить дальнейшее распухание.

## Deliverables

1. `docs/repository_shape.md` — короткий канонический документ со структурой репозитория.
2. `docs/deletion_policy.md` — правила удаления, архивации и сохранения материалов.
3. Один PR/commit серии “shape freeze”, в котором:
   - для каждой крупной директории зафиксирован её статус;
   - все неясные элементы получили решение: keep / move / archive / delete.

## Required decisions

### 1. Root-level inventory
Нужно пройти все top-level каталоги и файлы и для каждого указать:
- зачем он существует;
- является ли он first-class;
- кто его потребитель;
- должен ли он остаться в целевой форме repo.

### 2. Allowed top-level surface
Нужно явно зафиксировать, что целевая поверхность top-level ограничивается:
- build/release infrastructure;
- исходниками;
- тестами/benchmarks;
- canonical docs;
- примерами;
- scripts.

### 3. Removal policy
Нужно ввести четыре статуса:
- `keep` — активная часть репозитория;
- `move` — нужен, но лежит не там;
- `archive` — хранится только как справка и не участвует в навигации;
- `delete` — не должен оставаться в репозитории.

### 4. Navigation policy
Нужно определить один официальный маршрут чтения:
- `README.md`
- docs index
- canonical docs

Всё остальное не должно выглядеть как обязательная часть чтения.

## Work breakdown

### A. Составить полный inventory
- корень репозитория;
- `docs/`;
- `examples/` / `demo`;
- вспомогательные bat/md файлы;
- release/changelog infrastructure.

### B. Для каждого элемента принять решение
Запрещено оставлять статус “пока неясно”.
Для каждого спорного элемента нужно принять одно решение из четырёх: keep / move / archive / delete.

### C. Зафиксировать target shape
В `docs/repository_shape.md` описать:
- целевые директории;
- их роль;
- допустимое содержимое;
- что туда больше нельзя добавлять.

### D. Зафиксировать deletion policy
В `docs/deletion_policy.md` описать:
- когда документ удаляется;
- когда переносится в archive;
- когда допустим compatibility layer;
- когда временный файл обязан исчезнуть.

## Acceptance criteria

- Для каждого top-level элемента репозитория принят явный статус.
- У репозитория есть описанная target shape.
- У репозитория есть формальная deletion policy.
- Нельзя добавить новый “служебный” файл в корень без нарушения этого документа.
- После выполнения задачи дальнейшая компактификация делается уже по фиксированным правилам.

## Out of scope

- редизайн API;
- переписывание контейнеров;
- тестовая матрица;
- storage seams.

## Dependencies

Нет. Это стартовая задача пакета.
