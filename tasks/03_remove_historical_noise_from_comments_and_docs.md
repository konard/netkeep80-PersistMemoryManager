# Remove historical noise from comments and docs

## Purpose

Убрать из кода и документации **исторический и неинформативный шум**:
- ссылки на issue;
- narrative-комментарии про прошлые состояния;
- устаревшие пояснения;
- комментарии, которые пересказывают код вместо фиксации инварианта.

## Why now

Даже хороший код быстро выглядит раздутым, если вокруг него накапливаются:
- комментарии “из истории рефакторинга”;
- объяснения очевидного;
- устаревшие заметки;
- ссылки на issue вместо описания сути.

Это ухудшает читаемость не меньше, чем лишние строки кода.

## Deliverables

1. `docs/comment_policy.md` — короткий формальный policy-document.
2. Серия clean-up изменений в исходниках и docs:
   - удалены issue-ссылки из канонических docs и кодовых комментариев;
   - удалены устаревшие narrative-комментарии;
   - длинные комментарии переписаны в краткие invariant/design notes.
3. Обновлённый `CONTRIBUTING.md` с правилом комментариев.

## Work breakdown

### A. Ввести классификацию комментариев
Оставлять только комментарии четырёх типов:
- invariant;
- persistence contract;
- safety note;
- design note.

Всё остальное подлежит удалению или переписыванию.

### B. Удалить issue-oriented comments
Из кода и канонических docs убрать:
- `Issue #...`
- `TODO for issue #...`
- `implemented in #...`
- “оставлено после issue ...”

История уже есть в Git и PR.

### C. Удалить stale comments
Удалить или переписать комментарии, которые:
- противоречат текущему коду;
- описывают уже несуществующие варианты;
- ссылаются на старый allocator-first mental model там, где уже принят forest model;
- обещают “временное” решение без срока.

### D. Сжать многословные комментарии
Если комментарий длиннее, чем код под ним, он должен пройти отдельную проверку.
По умолчанию длинный narrative удаляется, а полезная часть превращается в 1–3 строки invariant note.

### E. Очистить docs от ссылочного шума
В canonical docs убрать:
- хронологии;
- перечисления закрытых issue;
- разделы “реализовано в версии ...”.

## Acceptance criteria

- В canonical docs нет ссылок на issue как части нормативного текста.
- В кодовых комментариях нет ссылок на issue и историю рефакторинга.
- Оставшиеся комментарии описывают только инвариант, риск или причину решения.
- Комментарии больше не конфликтуют с текущим состоянием кода.
- `CONTRIBUTING.md` задаёт правило, которое предотвращает повторное засорение.

## Out of scope

- удаление кода;
- тестовая матрица;
- storage seams.
