# Каталог требований PMM

Каждое требование оформлено как заголовок уровня `##` с идентификатором в формате
`ttt-xxx`, где:

- `ttt` — тип требования в нижнем регистре (например, `br`, `ur`, `feat`, `fr`,
  `dr`, `if`, `qa`, `con`, `sys`, `asm`, `dep`, `ac`, `rule`);
- `xxx` — номер требования начиная с `001`. Категория (если применима) встроена
  в идентификатор: например, `qa-rel-001`, `qa-thread-002`.

Идентификатор является якорем для трассировки между md-файлами и между md и
исходным кодом/тестами:

- ссылка на требование в другом файле: `[fr-001](05_functional_requirements.md#fr-001)`;
- ссылка из исходного кода или тестов: см. PMM anchor-комментарии формата
  `/*\n## req-fr-001\n*/`, см. `scripts/check-include-anchor-comments.sh`.

## Правила трассируемости

1. Каждое требование уровня `##` в md-файле имеет идентификатор в формате
   `ttt-xxx` или `ttt-<sub>-xxx` (для `qa`).
2. Каждое требование должно ссылаться (хотя бы по одной из категорий)
   на нижележащие требования, на исходный код или на тесты:
   - блок `Реализуется в:` перечисляет дочерние требования и/или файлы исходников;
   - блок `Проверяется в:` перечисляет критерии приемки и/или тесты.
3. Если требование реализует вышестоящее требование, оно явно указывает это в
   блоке `Реализует:` со ссылкой на якорь родителя.
4. В исходном коде PMM-anchor-комментарий формата `/*\n# anchor-name\n*/`
   ссылается на требования в комментариях в виде `req:fr-001`,
   `req:rule-009` (см. `scripts/check-requirements-traceability.py`).
5. В тестах рядом с anchor-комментарием размещается ссылка на проверяемые
   критерии приемки/требования в формате `req:ac-001` / `req:fr-021`.

## CI-проверка

Скрипт `scripts/check-requirements-traceability.py` валидирует трассируемость:

- структура заголовков и идентификаторов;
- наличие хотя бы одной нижестоящей ссылки (требование, исходник, тест);
- разрешимость всех внутренних ссылок (md-якоря).

Скрипт запускается в workflow `Docs Consistency` при изменениях в `req/`,
`include/` или `tests/`.
